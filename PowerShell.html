Key Concepts:

Getting Started with Microsoft PowerShell:

One of the most important topics covered by this segment of the MVA is the help system. PowerShell has many ways to get help on different cmdlets, parameters, aliases, etc. When first learning PowerShell, this system can save a lot of time searching the internet for an answer when instead you can get help directly in the console.

This segment of the course also covers the concept of the pipe. While the pipe is a critical aspect of PowerShell scripting, when writing new scripts for the first time I am careful to use them. Instead of using pipes to string together long lists of inputs processes and outputs, when scripting, I like to run each component and build onto it. This helps me to identify issues and understand each element of my script. Only then will I begin to combine cmdlets and processes into more complex items with pipes.

One very key concept the MVA introduces in the Objects for Admins section is the idea of filtering as close to the beginning of your script as possible. Several of the scripts I have used (including the hashing script I reference below) can take a long time to run. Filtering the selected dataset to only the needed information as early in the script as possible before any additional processing takes place can help optimize the operation.

Desired State Configuration:

I had to reset this section of the MVA several types because I did not have a working knowledge of what desired state configuration is, so developing the components of it in PowerShell didn’t make any sense. It wasn’t until I head one of the hosts of the lectures say “stop touching the box”, did this sort of scripting make sense.

While I’m sure that some companies use PowerShell to script DSC, I would assume more companies use a licensed tool to manage server configurations. However, I believe this portion of the MVA did an excellent job explaining the important of DSC and the overall dev ops mindset. They described the dev ops mindset as an attempt to keep the various spaces (production, test, development, QA) as in line as possible. To do this, developers should use common configurations were possible. Their goal is to house as many of the configs as possible in the common configuration scripts before moving to more specific scripts.

I topic I found extremely interesting in the Desired State Configuration portion of the MVA was the idea that system administrators should treat instances/servers as “cattle not pets.” Create reusing configuration scripts makes it easier to just delete and start over on a node when something goes wrong. This concept was very benefit for me to hear and understand before I attempted to create an OpenVPN server on Ubuntu. After misreading the directions (in all their many, many various pages). I remember this lecture and just deleted the server and started over. In that instance, having config files ready to script the creation of the new server would have made the start over process even easier.

PowerShell Use Cases:

Important note: all the following PowerShell use cases assumes the user of the script has at least read access to the files, file paths. If you try to run these scripts without the proper permission, you will get an error.

As an IT Analyst tasked with supporting HR, the primary way I use PowerShell is it gather information about Active Directory groups and group members. One of the major HR initiatives this year is to organize encrypted file shares. Below is a simple script to export members of an AD group to a csv:

Get-ADGroupMember -identity "ADGROUPNAME” | Select name, samaccountname | export-csv -path C:\groupmembers.csv

If you don’t have the AD Group name to find the members you can access it using the following:

Get-Acl \\path\to\the\folder |Select -Expand Access |Export-Csv C:\path\to\the\filename.csv -NoTypeInformation

The get-acl cmdlet is extremely helpful as it also shows you the file system rights (write, readandexecute, modify, fullcontrol, etc.) as well as describes if the file permissions for the particular item is inherited from another group.

I was also tasked with renaming all files within a folder. For this task, the new file name needed to eliminate the first characters of the original name that were generated by a system that could not easily and quickly be changed. This scripted when to a specific folder in a directory then selected all items in that directory that ended in .pdf. This was piped to a cmdlet that renamed the item, which in this case was a substring of the original name.

cd c:\path\to\files

get-childitem *.pdf | rename-item -newname { [string] ($_.name).substring(61)}

I have also used PowerShell to remove items from a file directory. During an automated process that generated 100s of .pdfs, 100s of .xml files were also created. The following script gets files within a specific directory with a .xml format then loops through those items deleting each one as it goes.

get-childitem -path 'c:\users\nroberts\desktop\TCTesting' *.xml | foreach {Remove-item -path $_.FullName }

During a data management discussion with the HR team, a concern arose about have multiple copies of the same exact files saved in file shares. In order to quantity this concern I used PowerShell to first create a hash (which served as a unique identifier) for each file, then compare the hashes to report on what documents existed in 2 places.

Get-childitem -file -recurse | Group-Object Length | Where-Object { $_.Count -gt 1 } | select -ExpandProperty group | foreach {get-filehash -literalpath $_.fullname} | group -property hash | where { $_.count -gt 1 } | select -ExpandProperty group | select hash, path | Out-File c:\path\to\file\duplicated_files.txt -width 510

A few important items to note: this script runs the get-childitem of the directory you are currently in so make sure you list to that directory before running. If you are comparing a large number of files this can take many hours to complete, especially when this data is encrypted. Once you have identified these duplicate files you can then use PowerShell to move or delete them in mass.

Online Course Evaluation:

The content provided by the Microsoft Virtual Academy was information rich but was tailored more for work IT professionals than students. Moreover, the segments of these courses usually an hour, making it harder to commit the information to memory. Finally, the webcast style of this MVA meant that the webcast meant that it included a lot of banter between the hosts, which in my opinion made the lectures longer than they needed to be. In summary, I think PowerShell is the most important skill I acquired with this course, but MVA was a poor delivery system for this content

My recommendation would be to instead use the PowerShell coursework from edX. This course basically covers the same information as the first half of the MVA (Getting Started with Microsoft PowerShell). While the content is the same, it is explained as though the user has never used PowerShell before and does not assume the user already in a server administration type role. The delivery of the content (short videos, articles, labs and quizzes) is preferable to the model offered by the MVA (long lectures, barely any substantive slides, and extremely challenging quizzes).
